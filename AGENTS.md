# AGENTS.md

このファイルは、このリポジトリで常に適用したい共通ルールを定義する。
役割別の振る舞いは `agents/` 配下のファイルで切り替える。

## 基本方針
- 変更前に、関連ファイルと既存実装の流れを読む。
- 推測で進める場合は、重要な前提だけ明示する。
- 既存の設計、命名、責務分割を優先し、不要な書き換えを避ける。
- 依頼範囲外のリファクタリングは、必要性が高い場合に限って提案を実施する。
- ファイル探索は Git 管理対象に限定せず、無視ルールを含めて確認すること（例: `rg --files -uu`）。
- 指示を実行する前に計画を提示すること。
- 計画に対して了承を得た場合のみ作業を開始すること。
- ライブラリ等のインストールはコンテナへ実行すること。ローカル環境へは絶対に実行しない。
- 開発環境はコンテナ上にある。ホスト環境へは絶対に関与しないこと。

## 実装ルール
- 既存ファイルに未関連の変更を混ぜない。
- 破壊的な操作や大きな仕様変更は、明示依頼なしに行わない。
- 新規コードは可読性を優先し、コメントは本当に必要な箇所だけに付ける。

## 調査と検証
- 問題調査では、再現条件、原因、影響範囲の順で整理する。
- 実装後は、可能な範囲でテスト、ビルド、lint などの確認を行う。
- 確認を実行できなかった場合は、その理由を最後に明記する。

## 出力スタイル
- 結論を先に述べ、説明は短く具体的にする。
- ファイル参照がある場合は、対象箇所が追いやすい形で示す。
- 長い説明より、次に何をすべきかが分かる回答を優先する。

## 役割ファイルの使い方
- レビュー担当: `agents/reviewer.md`
- 実装担当: `agents/implementer.md`
- 設計担当: `agents/planner.md`
- テスト担当: `agents/tester.md`
- レビュー担当: `agents/reviewer.md`

会話で役割を指定する例:

```text
AGENTS.md を共通ルールとして使い、追加で agents/reviewer.md を読んでレビュー担当として振る舞って。
```
