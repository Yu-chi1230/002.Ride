# Ride ポートフォリオ資料

## 1. プロジェクト概要

**Ride** は、バイク利用者向けに「車両コンディション管理」「短時間での最適ルート提案」「作品づくり支援」を一つの体験としてまとめた AI ライフログアプリです。  
忙しいライダーでも、限られた時間で安全確認から走行、記録作成まで完結できることを目指して設計しました。

- 対象ユーザー: 多忙なライダー、愛車管理を重視するオーナー、SNS 発信を行うクリエイター
- 提供価値:
  - Health: 愛車状態を記録し、メンテナンス判断を支援
  - Explore: 制約時間内で走行価値の高いルートを提案
  - Create: 撮影素材を印象的なビジュアルへ加工

## 2. 背景と課題設定

バイク趣味は、走る時間の確保、車両状態の把握、走行後の記録整理まで含めると意外に負荷が高く、特に仕事や家庭と両立するユーザーほど継続しにくい課題があります。  
Ride では、以下の 3 つの負担をまとめて下げることを狙いました。

- 乗る前: 車両状態に不安がある
- 乗っている間: 限られた時間で良いルートを選びづらい
- 乗った後: 記録や作品化に手間がかかる

## 3. 解決策

Ride は、1 回の利用体験を以下の流れでつなげています。

1. 出発前に Health でパーツ診断結果を確認する
2. Explore で時間制約付きのルートを生成する
3. Create で撮影した写真を加工し、走行体験を作品として残す
4. Settings や Contact で継続利用に必要な情報更新と問い合わせを行う

単機能アプリではなく、ライダーの一連の行動をつなぐ設計にした点がこのプロジェクトの特徴です。

## 4. 実装した主要機能

### 4.1 Health

- 手動 ODO 更新
  - 走行距離を入力し、車両情報とメンテナンス状態を更新
- 画像診断
  - 車両画像を送信し、AI 解析結果を表示
- 音声診断
  - エンジン音を送信し、診断結果を記録
- メンテナンス履歴連動
  - オイル交換距離などの更新時に履歴と状態を同期

### 4.2 Explore

- 最新ルート取得
  - Home から直近のおすすめルートを再利用可能
- ルート生成
  - 制約時間をもとにルートを作成し、関連データを保存
- ルート詳細表示
  - 所有者のみが詳細を参照できる権限制御を実装

### 4.3 Create

- 画像加工
  - 生成結果の表示とダウンロードまでを UI 上で完結
- 制作物保存
  - 生成履歴を記録し、利用ログとして残せる構成

### 4.4 Login

- Supabase Auth を使ったログイン導線を実装
- 未認証ユーザーと認証済みユーザーで遷移を制御
- オンボーディング未完了ユーザーを初期設定へ案内

### 4.5 Home

- 最新ルートやお知らせを表示し、各機能への起点として利用
- Explore で作成した最新ルートへの再導線を用意
- アプリ全体のハブ画面として利用継続を支援

### 4.6 Settings

- プロフィール更新、車両情報更新、オイル管理に対応
- 走行距離や前回オイル交換距離の整合性を保ちながら更新
- 背景画像変更など、利用環境に応じた表示設定も扱う

### 4.7 Onboarding / Contact

- オンボーディングでのプロフィール・車両登録
- Contact からの問い合わせ保存と Notion 連携

## 5. 技術構成

- フロントエンド: React 18 + Vite
- バックエンド: Node.js + TypeScript + Express
- DB / 認証 / ストレージ: Supabase
- ORM / DB アクセス: Prisma
- AI 活用:
  - Gemini を利用した解析系処理
  - 画像生成 / 加工系の機能設計
- 開発環境:
  - Docker ベースのローカル開発
  - frontend / backend / Supabase を分離した構成

フロントは `Login`, `Home`, `Health`, `Explore`, `ExploreGuide`, `Create`, `Settings`, `Contact`, `Onboarding` で画面分割し、バックエンドは認証、探索、健康管理、生成処理を API として分離しました。

## 6. 設計上の工夫

### 6.1 ユーザー導線の一貫性

機能を単発で並べるのではなく、「出発前」「走行中」「帰宅後」という利用シーンで役割がつながるように設計しました。  
これにより、機能追加ではなく体験全体の価値として説明しやすい構成になっています。

### 6.2 モバイル前提の UI 設計

画面デザインでは、車両写真が主役になるよう余白と視線誘導を重視し、暗めの配色と最小限の UI で没入感を出す方針を採用しました。  
また、iOS Safari のスワイプや bounce 挙動を抑制する実装を入れ、スマートフォン利用時の体験を安定させています。

### 6.3 実データ整合性を意識した更新設計

走行距離、前回オイル交換距離、メンテナンス履歴の関係をバックエンド側で検証し、矛盾した更新を拒否する実装にしました。  
単なる入力フォームではなく、運用データとして破綻しないことを優先しています。

### 6.4 外部連携失敗時の耐障害性

問い合わせ機能では、Notion 連携に失敗しても内部 DB 保存は継続し、一次受付を失わない設計にしています。  
外部サービス障害時も業務データを落としにくい点を重視しました。

## 7. 品質確認とテスト

`docs/tests/` 配下に仕様書と結果を整理し、単体・結合・システム・セキュリティの観点で確認を進めています。

### 7.1 主な確認結果

- 結合テスト
  - ログイン、プロフィール取得、Health、Explore、Create、Settings、Contact の主要導線が OK
- システムテスト
  - 2026-04-01 時点で主要シナリオは概ね OK
  - Google ログイン失敗時 UI、外部障害時 UI など一部未確認あり
- セキュリティテスト
  - 未認証アクセス、改ざん、IDOR、管理 API 権限は OK
  - 問い合わせ API のメタデータなりすましと添付ファイル公開設定に課題あり

### 7.2 テストから見えたこと

- 正常系だけでなく、入力不備や外部依存失敗も確認対象に含めている
- 不具合を見つけた箇所は、結果に NG 理由と再テスト条件を残している
- 品質面では一定の確認が進んでいる一方、セキュリティ設定や一部 UI 異常系には改善余地が残る

## 8. このプロジェクトで示せる強み

- 課題設定から UX ストーリーまで一貫して設計している
- React / Express / Supabase / Prisma を使ったフルスタック構成を組める
- AI 機能を単なる話題性ではなく、実際の利用導線に組み込んでいる
- テスト仕様と結果をドキュメント化し、機能の成立性を根拠付きで示している
- 外部連携の失敗時にもデータ保全を優先する設計判断ができる

## 9. 課題と今後の改善

- 問い合わせ API のメタデータ信頼境界を見直す必要がある
- 添付ファイル用 Storage バケットの公開設定は閉じる必要がある
- 外部サービス障害時の UI フィードバックを強化したい
- 背景画像保存など、一部仕様書と実装の差分を整理したい
- 画像診断や生成機能は、精度評価と再現性の検証をさらに深めたい

## 10. まとめ

Ride は、バイクユーザーの行動を「点」ではなく「流れ」で捉え、AI を使って体験をつなげることを目指したプロジェクトです。  
企画、UI、フロント実装、API、認証、DB、外部連携、テスト整理までを横断して扱っており、フルスタック開発とプロダクト思考の両方を示せる題材になっています。
