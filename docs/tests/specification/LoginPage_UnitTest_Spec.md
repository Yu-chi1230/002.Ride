# ログイン画面 単体テスト仕様書

## 1. 目的

`LoginPage` コンポーネントの単体テスト観点を整理し、認証 UI の基本挙動、入力バリデーション、Supabase 呼び出し、画面内状態変化を安定して検証できる状態にする。

対象実装:
- `frontend/pages/LoginPage.tsx`

関連境界:
- `supabase.auth.signInWithOAuth`
- `supabase.auth.signInWithPassword`
- `react-router-dom` の `useNavigate`

## 2. テスト対象範囲

本仕様書で扱う範囲:
- 初期表示
- メールログインフォームの入力制御
- バリデーションエラー表示
- Google ログインボタン押下時の処理
- メールログイン API 呼び出し時の状態変化
- エラー時メッセージ表示
- オンボーディング画面への遷移導線
- パスワード表示切替

本仕様書で扱わない範囲:
- Supabase 側の認証成否そのもの
- `AuthRoute` による `/home` への実遷移
- CSS 崩れ、アニメーション表現、粒子背景などの見た目検証
- ブラウザ間互換性試験
- E2E レベルの認証フロー通し確認

## 3. 前提条件

- テストフレームワークは `Vitest` + `@testing-library/react` + `@testing-library/user-event` を想定する。
- `LoginPage` は `MemoryRouter` 配下でレンダリングする。
- `supabase` モジュールはモック化する。
- `useNavigate` は呼び出し確認のためモック化する。
- `window.location.origin` は `http://localhost:5174` 相当で評価できる状態を前提とする。

## 4. モック方針

### 4.1 Supabase モック

対象:
- `supabase.auth.signInWithOAuth`
- `supabase.auth.signInWithPassword`

基本方針:
- 正常系は `error: null` を返す。
- 異常系は `error` を返すケースと `throw` するケースを分けて持つ。
- `signInWithPassword` の戻り値は `{ data: { user: ... }, error: null }` と `{ data: { user: null }, error: ... }` を使い分ける。

### 4.2 ルーター関連モック

対象:
- `useNavigate`

基本方針:
- 「アカウント作成はこちらから」押下時に `/onboarding` へ遷移要求したことだけを検証する。

## 6. 詳細テストケース

### LP-UT-001 初期表示

- 目的: 画面の主要 UI 要素がレンダリングされることを確認する。
- 事前条件: なし
- 操作:
  1. `LoginPage` をレンダリングする。
- 期待結果:
  1. `ride` ロゴが表示される。
  2. `あなたの愛車を、映画にする。` が表示される。
  3. `Google でログイン` ボタンが表示される。
  4. メールアドレス入力欄、パスワード入力欄、`ログイン` ボタンが表示される。

### LP-UT-002 初期状態

- 目的: 初期ステートが正しいことを確認する。
- 操作:
  1. 初期レンダリングする。
- 期待結果:
  1. エラーメッセージ領域は表示されない。
  2. パスワード入力欄の `type` は `password`。
  3. ログインボタンは活性状態。

### LP-UT-003 パスワード表示切替

- 目的: 表示切替ボタンが状態を反転させることを確認する。
- 操作:
  1. トグルボタンを押下する。
  2. 再度押下する。
- 期待結果:
  1. 1回目押下後、入力欄の `type` が `text`。
  2. 2回目押下後、入力欄の `type` が `password`。

### LP-UT-004 未入力バリデーション

- 目的: 必須入力不足時に API 呼び出しを防ぐことを確認する。
- 操作:
  1. メール未入力のまま送信する。
  2. パスワード未入力のまま送信する。
- 期待結果:
  1. `メールアドレスとパスワードを入力してください。` が表示される。
  2. `signInWithPassword` は呼ばれない。

### LP-UT-005 パスワード形式バリデーション

- 目的: パスワード形式チェックを確認する。
- 入力パターン:
  1. `abc123`
  2. `abcdefgh`
  3. `12345678`
- 期待結果:
  1. いずれも `パスワードは8文字以上で、英字と数字をそれぞれ1文字以上含める必要があります。` を表示する。
  2. `signInWithPassword` は呼ばれない。

### LP-UT-006 入力 trim

- 目的: 前後空白を除去して認証 API に渡すことを確認する。
- 操作:
  1. メール欄に `  test@example.com  ` を入力する。
  2. パスワード欄に `  abc12345  ` を入力する。
  3. 送信する。
- 期待結果:
  1. `signInWithPassword` は `test@example.com` と `abc12345` を受け取る。

### LP-UT-007 メールログイン正常系

- 目的: 正常ログイン時の API 呼び出しと状態遷移を確認する。
- 事前条件:
  - `signInWithPassword` は成功応答を返す。
- 操作:
  1. 正しい入力値で送信する。
- 期待結果:
  1. `signInWithPassword` が 1 回呼ばれる。
  2. 送信中に `ログイン中...` が表示される。
  3. 完了後に `ログイン` 表示へ戻る。
  4. エラーメッセージは表示されない。

### LP-UT-008 メールログイン認証失敗

- 目的: 認証失敗時のメッセージ表示を確認する。
- 事前条件:
  - `signInWithPassword` は `error` を返す。
- 操作:
  1. 正しい形式の値で送信する。
- 期待結果:
  1. `メールアドレスまたはパスワードが間違っています。` が表示される。
  2. ボタンは再度活性化される。

### LP-UT-009 メールログイン例外

- 目的: API 例外時に汎用エラーへフォールバックすることを確認する。
- 事前条件:
  - `signInWithPassword` が例外を送出する。
- 操作:
  1. 正しい形式の値で送信する。
- 期待結果:
  1. `エラーが発生しました。もう一度お試しください。` が表示される。
  2. ボタンは再度活性化される。

### LP-UT-010 ログイン中表示

- 目的: 非同期処理中の二重送信抑止を確認する。
- 事前条件:
  - `signInWithPassword` を未解決 Promise にする。
- 操作:
  1. 送信する。
- 期待結果:
  1. 送信ボタンが `disabled` になる。
  2. ボタン文言が `ログイン中...` に変わる。

### LP-UT-011 Google ログイン正常系

- 目的: OAuth 呼び出し内容が正しいことを確認する。
- 事前条件:
  - `signInWithOAuth` は成功応答を返す。
- 操作:
  1. `Google でログイン` を押下する。
- 期待結果:
  1. `signInWithOAuth` が 1 回呼ばれる。
  2. 引数に `provider: 'google'` を含む。
  3. `options.redirectTo` は `${window.location.origin}/home`。
  4. エラーメッセージは表示されない。

### LP-UT-012 Google ログイン失敗応答

- 目的: OAuth エラー応答時の専用メッセージ表示を確認する。
- 事前条件:
  - `signInWithOAuth` は `error` を返す。
- 操作:
  1. Google ログインボタンを押下する。
- 期待結果:
  1. `Googleログインに失敗しました。` が表示される。

### LP-UT-013 Google ログイン例外

- 目的: OAuth 例外時の汎用メッセージ表示を確認する。
- 事前条件:
  - `signInWithOAuth` が例外を送出する。
- 操作:
  1. Google ログインボタンを押下する。
- 期待結果:
  1. `エラーが発生しました。もう一度お試しください。` が表示される。

### LP-UT-014 エラークリア

- 目的: 再送信時に古いエラーが残留しないことを確認する。
- 事前条件:
  - 1 回目は認証失敗
  - 2 回目は成功
- 操作:
  1. 1 回目送信でエラー表示させる。
  2. 値を修正して再送信する。
- 期待結果:
  1. 2 回目送信開始時点で前回のエラーがクリアされる。
  2. 成功後はエラー表示が残らない。

### LP-UT-015 サインアップ導線

- 目的: 新規登録導線が正しい画面へ遷移することを確認する。
- 操作:
  1. `アカウント作成はこちらから` を押下する。
- 期待結果:
  1. `navigate('/onboarding')` が 1 回呼ばれる。

## 7. 実装時の補足

- `required` 属性だけではなく、React 側の独自バリデーション文言をアサート対象にする。
- `console.error` を呼ぶ実装があるため、テストでは必要に応じて `vi.spyOn(console, 'error').mockImplementation(() => {})` を使う。
- `GoogleIcon` の SVG 自体は単体テスト対象外とし、ボタンラベルで要素取得する。
- 成功時は本コンポーネント内で明示的な `navigate('/home')` をしていないため、単体テストでは「成功 API 呼び出し完了まで」を責務境界とする。

## 9. リスク・注意点

- 現実装ではメール形式の詳細チェックはブラウザの `type="email"` に依存しており、コンポーネント独自バリデーション対象ではない。
- ログイン成功後の遷移は `AuthRoute` と `AuthContext` 側に依存するため、本仕様書では責務外として切り離している。
- 非同期状態更新の確認では `findByText` や `waitFor` を用いて一時状態と終了状態を分けて確認する。
