# DBテーブル定義の確認と提案

作成いただいた `docs/DB` 配下のSQLファイル（`DDL_Create.sql`, `DDL_Explore.sql`, `DDL_Health.sql`）を拝見しました！
MVPドキュメントの要件（AI診断、ルート探索、客観的色覚補正、ナレーション生成など）を網羅した非常に綺麗でスケーラブルな設計になっています。

全体の構成は完璧に近いですが、**アプリのUXをさらに向上させ、データの不整合を防ぐため**に、以下のテーブルへの「カラム追加」および「トリガー（自動処理）の追加」を提案します。

---

## 1. カラムの追加提案

### ① `vehicles` テーブル（DDL_Health.sql）
現在の走行距離（ODO）と前回のオイル交換距離は管理されていますが、他の消耗品の基準値も持っておくと今後の拡張が楽になります。
* **[提案]**: `last_chain_maintenance_mileage` (integer) - 前回チェーン清掃/交換時の走行距離
* **[提案]**: `last_tire_change_mileage` (integer) - 前回タイヤ交換時の走行距離
* **[理由]**: AI画像診断（タイヤ、チェーン）の結果を受けて「何km走ったから劣化しているのか」のロジックに利用するためです。（※都度 `maintenance_history` を集計計算するよりパフォーマンスが良いです）

### ② `health_logs` テーブル（DDL_Health.sql）
* **[提案]**: `status` (ENUM: 'pending', 'analyzing', 'completed', 'failed')
* **[理由]**: 音声や画像の「AI解析中」は数十秒かかる可能性があります。フロントエンドで「解析中スケルトンUI」を出すため、バッチ処理や非同期処理を見据えたステータス管理カラムがあると便利です。

### ③ `routes` テーブル（DDL_Explore.sql）
* **[提案]**: `actual_duration_minutes` (integer) - 実際の走行にかかった実測時間
* **[理由]**: ユーザーが「1時間ルート」を選択しても、実際に何分かかったかの誤差を蓄積することで、次回以降のルート算出AIの精度（ユーザーのペース配分）を学習させるためです。

---

## 2. PostgreSQL（Supabase）トリガーの追加提案

システム側で自動でデータ整合性を保つためのスマートなDBトリガーです。

### ① 走行距離（ODO）の更新運用
* **[現状]**: ユーザーによる手動入力を基本とします。
* **[理由]**: OCRの誤認識リスクを排除し、ユーザーが確実に現在の数値を把握・管理できるようにするためです。将来的にAPIによる自動取得を検討する余地は残します。

### ② 【推奨】メンテナンス履歴の自動記帳トリガー
* **[トリガー内容]**: `vehicles` テーブルの `last_oil_change_mileage` などの特定カラムが更新された場合（手動またはAI OCR検知により）、自動的に `maintenance_history` テーブルへ「オイル交換 (自動追記)」としてレコードをINSERTする。

## どう進めますか？

現状のテーブル定義のみでも「ride」v1.0の要件は十分に満たせます。
上記の追加提案について、以下からご希望の方針を教えてください！

1. **提案を取り入れる**: 上記のカラムやトリガーを追記した `DDL_Update.sql` をこちらで作成します。
2. **そのまま進める**: まずは現状のDB定義のままで、フロントエンド（Health: Tool 画面など）の実装を再開します。
